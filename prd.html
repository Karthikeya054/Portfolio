📄
Product Overview

UniConnect is an internal multi-university operations platform built for NxtWave's university partner program. It unifies student communication, task management, operational reporting, academic workflows, budget tracking, and role-based access into a single system — replacing a fragmented web of personal Gmail accounts, WhatsApp groups, and shared Google Sheets.

The platform is designed for university operations staff, faculty, and leadership across 18 partner institutions, with the goal of replacing manual effort with automated, structured workflows that surface real-time visibility to leadership without adding burden to front-line teams.

02
⚠️
Problem Statement

University operations across 18 NxtWave partner institutions were managed through entirely informal channels. The core pain points surfaced through 40+ user interviews:

  • Bulk email took 2–3 hours per campaign. Faculty manually composed and sent individual batches from personal Gmail accounts, with no delivery tracking, no acknowledgment confirmation, and no way to resume if the process failed mid-send.
  • Zero leadership visibility. Department heads had no real-time view into task completion, team activity, or performance across campuses. Status updates happened via informal phone calls.
  • Exam paper creation consumed 4+ hours per paper. Faculty manually selected questions, formatted papers, assigned Bloom's taxonomy levels — an entirely manual and repetitive process with no question bank or reuse mechanism.
  • Budget proposals had no audit trail. Approval chains happened over email. Post-event reconciliation had no structured process, creating financial accountability gaps.
  • Meeting outcomes were lost. Decisions made in meetings were captured in personal notes and never systematically followed up on, causing recurring re-discussions and slipped action items.
  • Faculty data entry consumed 6+ hours per week per person. Attendance sheets, marks entry, and performance reports were all manual, redundant, and error-prone.
Root Cause
The root cause was not tool absence — it was fragmented informal systems creating invisible operational overhead that scaled linearly with each new university added, with no feedback mechanism for leadership to see the scale of inefficiency.

  • Enable scale from 1 to 18+ universities without custom development per institution
  • Eliminate manual faculty data entry for attendance, marks, and performance sheets

  • Not designed for public SaaS distribution in Phase 1 — internal deployment only
  • Not a communication tool for students communicating with each other

Key Technical Decisions & Rationale
BullMQ + Redis for Email Queue
Decision: Process all bulk email sends asynchronously through a Redis-backed job queue.

Rationale: Synchronous sends would block the UI for minutes during large campaigns. BullMQ provides durable job persistence (survives restarts), retry logic, and resume capability — so an interrupted 500-student send picks up from the last completed row, not from scratch.

Trade-off: Adds Redis infrastructure dependency. Accepted — reliability at this scale requires it.
University-Scoped Data Model
Decision: All tables include a university foreign key; all queries are filtered by university at the API middleware layer.

Rationale: Chosen over a shared table model to enforce complete data isolation. A query-layer approach would risk isolation bugs under complex joins. Middleware enforcement is simpler to audit and test.

Trade-off: Slower initial onboarding (university config must be set before any data entry). Accepted — data leakage at 18-university scale would be catastrophic.
Auto-Aggregation over Manual Reporting
Decision: The Ops Dashboard auto-aggregates data from all modules. The 2-minute daily form adds qualitative context — it does not replace auto-aggregated data.

Rationale: If leadership visibility required teams to fill yet another form, adoption would fail. By making visibility a byproduct of normal system use, it adds zero burden to front-line teams while giving leadership more accurate data than self-reported forms alone.

Product Overview

A secure, QR-based digital attendance tracking system for NxtWave bootcamps. Replaces paper-based attendance entirely with a real-time tablet-scanner system: each student carries a unique QR code, authorized scanner tablets at the venue read them, and attendance is recorded instantly. The system is built with a tamper-proof token architecture, cross-platform support (iOS Safari + Android Chrome), and full admin tooling for coordinators.

02
⚠️
Problem Statement
  • Paper attendance created post-session overhead. Sheets collected at the door had to be manually keyed into a spreadsheet — an error-prone process with no real-time visibility during the session itself.
  • Naive QR implementation would be exploitable. A URL-encoded QR code can be opened in any browser — students could mark themselves present without attending by simply visiting the URL from anywhere.
  • No device management existed. Multiple tablets acted as scanners with no way to authorize, revoke, or track which device recorded which entry — a significant accountability gap.
  • iOS Safari crashed on camera access. After initial deployment, iPads running Safari threw a null srcObject error when attempting to activate the camera — blocking the scanner entirely on Apple hardware, which was the primary scanner device.
03
🎯
Goals & Non-Goals
Goals
  • Eliminate paper attendance sheets entirely — zero paper after deployment
  • Prevent proxy/cheating attendance through tamper-proof token architecture
  • Provide real-time attendance visibility to session coordinators during the event
  • Support both iOS Safari and Android Chrome without crashes or workarounds
  • Give admin full device management (authorize, revoke, track)
Non-Goals
  • Not a student identity or authentication system
  • Not designed for daily classroom attendance — bootcamp events only
  • Not integrated with an external LMS in v1
  • No student-facing app — students show a static QR code only

Key Technical Decisions
Raw Token vs URL Encoding for QR
Options evaluated: URL-encoded QR (easy to implement) vs Raw opaque token QR (slightly more complex).

Decision: Raw token encoding. When a QR encodes a URL, scanning it opens a browser tab — any student can visit that URL from anywhere and potentially exploit it. Raw token encoding means the QR contains meaningless data to a browser; the scanner app alone can interpret and validate it server-side.

Result: Replay attack vector eliminated at the protocol level — no security logic required beyond token validation.
iOS Safari Camera Bug — Root Cause & Fix
Symptom: null srcObject error on camera activation in Safari on iPads.

Root Cause: The video element was conditionally rendered (mounted/unmounted based on state). The camera activation code ran before the element was mounted to the DOM, so getUserMedia had no target element.

Fix: Video element always mounted; toggled by CSS visibility, not DOM presence. Camera starts by calling getUserMedia on the already-mounted element.

Result: Zero crashes on all tested iOS devices post-fix. No workaround required — root cause addressed.
Device Pairing Model
Decision: Each scanner tablet goes through a one-time pairing flow: admin generates a setup QR per device → tablet scans it → device is registered with a unique deviceKey → admin explicitly authorizes it in the portal before it can submit.

Rationale: Prevents any unauthorized device (personal phones, unregistered tablets) from submitting attendance records. All submissions carry device labels — creating accountability at the device level, not just the student level.

Product Overview

A zero-login, self-serve student results portal built for the Makers Conclave hackathon at NIAT. Students enter their ID and instantly see their phase-wise scores, pass/fail status, and feedback — eliminating all manual result queries to event coordinators. Built on top of the existing Google Sheets data maintained by event staff, with an in-memory cache to handle peak traffic at results announcement.

02
⚠️
Problem Statement

During and after Makers Conclave results announcements, students repeatedly contacted event coordinators to ask for their scores — "What was my Phase 2 score?", "Did I qualify?", "Can you check my ID?" — creating significant coordinator overhead at exactly the moment when coordinators were busiest managing the event itself.

  • Coordinator overload: 50+ manual result queries per hour after results were announced, each requiring a coordinator to open the spreadsheet, search the ID, and reply individually.
  • Student frustration: Delays in getting results created anxiety and eroded trust in the event organization — a fixable UX problem, not an organizational one.
  • No structured data access: Results existed in a Google Sheet maintained by coordinators, but no structured student-facing interface existed to query it.

  • Eliminate all manual result queries to coordinators post-results announcement

  • Read from the existing Google Sheet — no new data entry burden on coordinators
  • Be embeddable into the existing NIAT website via iframe in under 5 minutes
Non-Goals
  • Not a score entry or management system — read-only access to existing data
  • Not a full event management platform
  • No admin portal for modifying scores (coordinators continue using Google Sheets)
  • No persistent user accounts or profiles

Key Technical Decisions
Google Sheets as Data Source (No New Backend)
Decision: Read directly from the Google Sheet coordinators already maintain via the Sheets API, rather than building a separate database.

Rationale: Coordinators already update the spreadsheet as scores are finalized — asking them to mirror data into a new system would create adoption friction and maintenance burden. The best product decision was: extend the existing workflow, don't replace it.

Trade-off: Google Sheets API has rate limits. Mitigated with in-memory cache.
Zero-Login Design
Decision: Students look up results by student ID only — no authentication, no account, no session.

Rationale: Results are not sensitive enough to require authentication. Any login friction reduces usage and increases coordinator load (students who can't log in will message the coordinator instead). Zero-login means every student can access results in under 10 seconds, from any device.
In-Memory Cache over Redis
Decision: Used Node.js in-process in-memory cache with a 5-minute TTL rather than a Redis instance.

Rationale: This was a short-lived event portal (5 days). Redis adds infrastructure overhead and cost unjustified for a feature at this scale. In-memory cache adequately handles peak traffic bursts at announcement time while keeping the system simple and deploy-ready in minutes.

Product Overview

A single-executable Python automation tool that reads faculty marks from a CSV file and automatically enters them into CDU's university management portal. Designed to require zero technical knowledge from faculty — one double-click to launch, with live progress feedback, error logging, and resume capability. Built and delivered in under half a day.

02
⚠️
Problem Statement

CDU faculty were required to enter marks for every student individually into the university's web-based management portal. The portal had no bulk import functionality — every student required manual navigation: open student record → select subject → enter marks → save → next student.

  • Time cost: 2–3 hours per subject per faculty member — for a repetitive, zero-cognitive-value task that added nothing beyond data transfer.
  • Error rate: Manual entry at scale introduces keystroke errors (transposed digits, wrong subject columns) that are hard to detect until students raise disputes.
  • No resume capability: If the portal session expired or a browser crash occurred mid-entry, faculty had to determine where they stopped and restart from there — no checkpoint, no log.
  • Zero visibility into progress: Faculty had no way to know how far through the class roster the upload had progressed without manually counting completed entries.
03
🎯
Goals & Non-Goals
Goals
  • Reduce marks entry time from 2–3 hours to under 30 minutes per subject
  • Eliminate manual keystroke errors through automated, CSV-driven entry
  • Require zero technical setup from faculty — no terminal, no Python installation, one double-click
  • Provide live progress visibility so faculty know the tool is working
  • Resume from checkpoint if the tool or session is interrupted mid-upload
Non-Goals
  • Not a portal replacement — works within the existing CDU portal UI
  • Not intended for other universities' portals in v1 (CDU-specific automation logic)
  • No cloud component — fully local, self-contained executable, no internet dependency beyond CDU portal access

I want the tool to resume from where it stopped if my laptop dies or the session expires mid-upload, so I don't have to figure out where it stopped and restart from scratch.
As a Department Coordinator
I want a summary at completion showing how many entries succeeded, how many were skipped, and why, so I can verify the upload was complete without re-checking the portal row by row.

Key Learning
The entire value of this tool came from ruthless constraint on scope. Faculty didn't need a UI framework, a database, or a web app — they needed a script that did exactly one thing perfectly: read a CSV and fill a form. Saying no to scope creep (GUI, multi-subject batch, reporting dashboard) kept the build to half a day and the delivery to the same afternoon.
